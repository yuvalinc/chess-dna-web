Chess DNA — Architecture & Onboarding Guide

01

Elevator pitch

2 min read

Chess DNA imports a player's chess.com / Lichess games, analyses them locally with Stockfish, then surfaces a personalised skill profile, weakness patterns, and AI-written insights about how they're really playing.

What the user sees

A radar chart of 8 chess skills, a rank tier (Pawn → King), recent-games list with per-move analysis, weakness patterns (e.g. "blunders in king-side attacks"), AI-written insights, and a TimeMachine that replays past opponents as puzzles.

What happens under the hood

PGNs are fetched and parsed client-side, then a singleton Stockfish WASM Web Worker (depth 18) analyses every move. Aggregations feed a pattern engine that detects ~19 themes across the last 50 games.

Where AI fits

Claude (sonnet-4) generates Insight cards from real analysis data — the only AI-generation surface in the app. Calls are server-proxied through a Base44 function so the API key never reaches the browser.

02

Quick start — running locally in 15 minutes

setup

No separate backend to start. Base44 is hosted; you run the React app locally and point it at production.

Prerequisites

  • Node 20+ (we develop on macOS, but Linux works)
  • An invitation to the Base44 project (admin is yuval.inc@gmail.com)
  • Optional: a personal Anthropic API key to bypass the shared VITE_FALLBACK_CLAUDE_KEY

Run it

git clone <repo> Chess-dna
cd Chess-dna
npm install            # postinstall copies stockfish WASM into public/
npm run dev            # Vite at http://localhost:5173

What to try first

  1. Sign in (Base44 redirects to its login; JWT lands in localStorage).
  2. In Settings, enter a chess.com username and click Import games.
  3. Watch DevTools console for [Chess DNA] and [Chess Tutor] logs as Stockfish chews through the games.
  4. Once analysis finishes, the Overview page shows the skill radar, and Patterns / Lessons / Exercises unlock.
▶ Dev shortcut DevModeToggle in the corner lets you simulate auth states and journey stages without re-importing games.

Tech stack

overview

All of this is in package.json — listed here so you can size things up before opening files.

Frontend runtime

React 19 · React Router 7 · TypeScript 5.9 (strict) · Vite 7. Tailwind CSS 4 via @tailwindcss/vite, with a custom chess theme keyed off a [data-theme] attribute on <html>.

Backend baas

Chess engine wasm

AI external

Claude (claude-sonnet-4-20250514) for Insight generation, proxied through Base44's claude-proxy function. Gemini / OpenAI client files still ship in the bundle but the router throws for anything except Claude.

Visualisation

Recharts for the radar, line, area, and composed charts. html2canvas + mp4-muxer for the share-card export. @xyflow/react powers the /graph dev visualisation.

Tooling

ESLint 9 + typescript-eslint + React-hooks plugin. No test framework yet — verification is manual via dev server. A prebuild script (check-deploy-safety.mjs) blocks worktree deploys when main is dirty.

One-page-per-route

src/pages mirrors the router 1:1. Adding a new screen = new file in pages + a <Route> in App.tsx. No code-splitting yet; everything ships in one bundle.

Cross-cutting state in contexts

Only 3 contexts exist (Auth, ChessData, Tutorial). ChessDataContext is the heavyweight one (1,260 LOC, 40+ derived values).

Engine / AI / patterns are leaf modules

They expose plain functions or singletons. They do not import React. Everything stateful is glued by contexts and route-level pages.

Types live in one place

src/shared/types/ is the contract. Adding a field to a Base44 entity means updating both the JSONC schema and the TS type — they are not auto-generated.

08

Core data pipeline

7 stages

From "user pastes a PGN" to "weakness pattern shows up on the radar". Seven discrete stages, three of which write to Base44.

1 · Importchess.com · Lichess · paste PGN
2 · ParseparsePgnToGameRecord via chess.js
3 · PersistGame entity → Base44
4 · AnalyzeStockfish d18 · classify · phase · tactical
5 · PersistAnalysis entity (JSON)
6 · ComputePatterns (19 themes) · 8-dim skill · benchmarks
7 · PersistPattern + PatternSnapshot

Important properties

  • Batch mode — isBatchMode() in analysis-pipeline.ts suppresses incremental re-renders during bulk imports. Use it whenever you're analysing > 1 game in a row.
  • Event-driven progress — analysisEvents emitter pushes progress updates; UIs subscribe instead of polling.
  • Queue, don't call directly — always use queueForAnalysis() from ChessDataContext. Calling runBatchAnalysis directly bypasses the queue and can spawn duplicate work.
  • Windowed — pattern + skill computation looks at the last 50 games by default (DEFAULT_WINDOW_SIZE in constants).
  • Singleton patterns — there is exactly one Pattern row per user (current state); history lives in PatternSnapshot rows.

Move classification

chess.com-style "Expected Points" model. We compute win-chance loss between the played move and Stockfish's top choice, then bucket:

BucketWin-chance lossDefault score (0–99)
brilliant / great / best0.00 — tagged via sacrifice / alternative-gap heuristics99
excellent≤ 0.0275
good / book≤ 0.0550 / 65
inaccuracy≤ 0.1015
mistake≤ 0.203
blunder> 0.200
forced—excluded from the average

All thresholds live in src/shared/constants.ts. Changing a number here ripples through the entire scoring system — bump the schema version too.

09

AI generation router

claude · insight-only

After the May refactor only one AI surface remains: the Insight generator + per-move explanation streams. Lessons, exercises, training plans, and podcast scripts are all gone. The router shape stayed because it was small and still has a job.

Two consumers (insights + per-move explanations) share one router, one provider, one proxy.

Active consumers

  • Insight generator (src/ai/insight-generator.ts) — turns recent GameSummary[] + WeaknessPattern[] into Insight cards. The only "card" generation surface left.
  • Move explanation streams — when you click a move in GameDetail, sendWithFallbackStream() generates an explanation chunk-by-chunk. Cached in SessionStorage (explanation-cache.ts) so re-clicking the same move is free.

Other notable hops

  • Token tracking — every successful Claude call updates the local tokenUsage (localStorage) and syncs to UserPreferences.
  • Removed validation — stockfish-validator.ts was deleted with the exercise generator. If you re-add AI-generated puzzles, you'll need a new validator.
  • Best-effort analytics — every AI request emits an ai_request event; failures here never block generation.
  • OpenAI / Gemini clients still ship in the bundle but createProvider() throws for anything other than Claude. Marginal dead weight — left for easy re-enable.

The useEntity contract

// All three return [data, loading, error, refetch]
useEntityList<T>(entityName: string, query?: object)
useEntityById<T>(entityName: string, id: string | null)
useSingletonEntity<T>(entityName: string)

Hooks branch on guest vs auth, retry on 429 with exponential backoff, and cache results in module-level maps. The hook count must stay stable across renders — don't put one behind a conditional.

⚠ Schema discipline Base44 silently drops unknown fields. If a save "succeeds" but the data didn't show up, the field probably isn't in the entity JSONC. Update the schema in the Base44 UI and commit the JSONC.
⚠ 5,000-row ceiling entity.list() caps at 5,000 records. For anything that could grow past that (games, analyses, events) you must paginate or query with filters.

Storage architecture

3 tiers

Three tiers: Base44 (durable, cross-device), localStorage (durable, single browser), in-memory + SessionStorage (ephemeral). IndexedDB is gone — it only held audio sessions, which were removed.

Server-side · Base44 durable

All 15 entity schemas + the CLAUDE_API_KEY server env secret. RLS scoped per user. The only place that survives reinstalling on a new device.

localStorage durable

JWT, guest-mode games, token usage, tutorial step, theme & locale, schema version. Beware the iOS Safari ~5 MB quota — that's why we cap snapshots at 200.

SessionStorage / memory ephemeral

Move-explanation cache (explanation-cache.ts) — re-clicking a move doesn't re-call Claude. chess.com avatars in an in-process Map. Both reset on tab close.

Storage keys live in STORAGE_KEYS in src/shared/constants.ts. When bumping a schema, also bump meta:schema_version and add a migration in settings-store.ts.

Stockfish & analysis

depth 18 · singleton

One singleton Web Worker handles all engine work. Breaking it breaks every analysis flow in the app.

Singleton client

import { StockfishClient } from '@/engine/stockfish-client';
const sf = StockfishClient.getInstance();
const result = await sf.analyzePosition(fen, { depth: 18 });

Pipeline modules

FileRole
engine/stockfish-client.tsSingleton wrapper around the Web Worker, UCI protocol
engine/uci-parser.tsParses raw info ... lines
engine/game-analyzer.tsOrchestrates per-move analysis (top moves, eval, pv)
engine/analysis-pipeline.tsBatching, progress events, persistence
engine/eval-classifier.tsWin-chance loss → quality bucket
engine/phase-detector.tsMaterial-weighted opening / middle / end phase
engine/tactical-detector.tsForks, pins, hanging-piece heuristics
engine/bot-mover.tsPlays moves against the user (used by TimeMachine)

The previous ai/stockfish-validator.ts module was deleted with the exercise generator. If you re-introduce AI-generated puzzles, plan on rewriting it.

Performance notes

  • Depth 18 is a deliberate tradeoff — deep enough for accurate classification, fast enough that 50 games analyse in ~2–4 minutes on a modern laptop.
  • Validation runs at depth 16 (cheaper) with a 50cp tolerance — exercises only need to be "good enough", not engine-perfect.
  • Worker is created lazily on first use, persisted for the rest of the session. Don't new StockfishClient() — always go through .getInstance().

Skill profile & rank tiers

8 dimensions · 6 tiers

The radar's 8 axes and the King/Queen/Rook/… rank are produced by skill-calculator.ts. Weights sum to 1.0.

DimensionWeightWhat it measures
Tactics0.18Critical-position accuracy + missed tactics
Defense0.15Performance when under attack / worse eval
Calculation0.15Accuracy on long-PV positions
Positional0.13Quiet-move quality, pawn structure
Endgame0.13Late-game accuracy (low material)
Openings0.12Book conformity + early-game accuracy
Time Management0.10Time-pressure blunder rate
Resilience0.07Recovery from worse positions

Rank tiers

♙ Pawn · 0–29 ♘ Knight · 30–44 ♗ Bishop · 45–59 ♖ Rook · 60–74 ♕ Queen · 75–89 ♔ King · 90–99

Defined in src/patterns/rank-tiers.ts. getTierForScore(overall) returns the tier object (label, colour, icon).

Opponent-strength adjustment

Skill scores are normalised against a 1200-elo baseline opponent. Beating a 2000 weighs more than beating an 800 — see windowed-profile.ts.

Fallback for broken joins

skill-calculator.ts has fallback logic for when Game ⨝ Analysis joins are incomplete (e.g. analysis dropped mid-batch). Don't simplify this away — historical accounts have orphan analyses.

Patterns & benchmarks

19 themes · last 50 games

The pattern engine looks at the most recent 50 games and tags up to 19 themes — things like "loses pieces in king-side attacks" or "panic blunders under 30 seconds".

  • Minimum games — PATTERN_MIN_GAMES = 3. Below this, patterns stay empty (UI shows an onboarding message).
  • Theme catalogue — see src/patterns/pattern-engine.ts. Each theme is a pure function over the windowed game set; adding one is a matter of writing the function + registering it.
  • Benchmarks — score-benchmarks.ts compares the user's per-dimension scores against rough rating cohorts (800 / 1200 / 1600 / 2000).
  • Snapshot history — every meaningful change writes a PatternSnapshot; MAX_PATTERN_SNAPSHOTS = 200 prunes older ones to stay under the iOS storage cap.

Cross-cutting concerns

conventions
Auth & access

Base44 JWT in localStorage; guest mode (pre-auth play, IndexedDB persistence); beta whitelist via BetaTester; admin role via email match. auth.me() may 401 even when the token is fine — don't gate CRUD on it.

AI routing & safety

One proxy, one provider, two consumers (Insights + move explanations). Token tracking is per-user; the shared VITE_FALLBACK_CLAUDE_KEY lets brand-new users try the app without bringing their own key.

Performance

Stockfish singleton; useMemo wraps every heavy derived value in ChessDataContext; batch mode suppresses re-renders during imports; per-move explanations stream chunk-by-chunk. Bundle is single-chunk — split later if perf becomes an issue.

Theming

Tailwind 4 with CSS vars on [data-theme]. Custom palette: chess-surface, chess-text, chess-accent, plus per-quality colours. 8 board themes. Mobile-first.

i18n

English / Español / עברית. Hebrew flips to RTL. Strings live in src/i18n/locales/{en,es,he}.ts; locale is stored in UserPreferences.

Analytics

Custom AnalyticsProvider writes to the AnalyticsEvent entity. Best-effort: failures never block UX. Admin reads aggregates at /analytics.

Share cards

Rendered with html2canvas. Do not use Tailwind CSS variables in components/share/* — html2canvas can't resolve them. Use SHARE_COLORS hex constants.

Prebuild safety

scripts/check-deploy-safety.mjs aborts the build when run from a worktree while main has uncommitted changes. Do not bypass it — twice it has caught wiped-prod deploys.

Critical files — do not break these

high blast radius

High blast radius. Read them before changing them. Most accidents in this codebase trace back to one of these.

src/contexts/ChessDataContext.tsx

1,260 LOC, 40+ derived values, event listeners with refs. All hooks must come before any early return (React error #300). Touch carefully.

src/pages/GameDetail.tsx

The most hook-heavy page. Every useState / useEffect / useMemo / useCallback must precede every conditional return.

src/engine/stockfish-client.ts

Singleton WASM worker. Break it and every analysis path breaks with it. Test imports + game detail + exercises after any change.

src/patterns/skill-calculator.ts

Profile computation. Keeps fallback logic for broken Game⨝Analysis joins — historical accounts depend on it.

src/shared/constants.ts

Single source of truth for thresholds, endpoints, regexes, storage keys. Changes ripple — bump the schema version when shifting bucket scoring.

src/hooks/useEntity.ts

Base44 entity hooks with guest/auth branching. Hook count must stay stable across renders. Used by literally every page.

src/ai/ai-router.ts

Provider fallback & streaming. Keep both sendWithFallback and sendWithFallbackStream in sync.

src/ai/prompt-builder.ts

~195 LOC (down from 685 after the May cleanup) — system prompt + Insight + explanation templates. Pulls latest revisions from the AIPrompt entity.

Build & deploy

prod only

Exactly one production environment: chess-dna-fdd5fbde.base44.app. There is no staging.

# From /Users/yuval/Chess-dna (NOT a worktree)
npm run build                  # prebuild safety + tsc -b + vite build → dist/
npx base44 site deploy -y      # uploads dist/ to Base44 hosting
  • Verify the bundle hash changes. dist/assets/index-*.js is content-hashed. If unchanged, the build didn't pick up your edits.
  • Don't trust tsc -b exit code. Incremental cache bugs exist — check that dist/ exists and the hash advanced.
  • Cache-bust after deploy. Hard-refresh and check the network tab for the new bundle filename. Mobile Safari is the worst offender for caching old bundles.
✕ NEVER deploy from a worktree Main almost always has uncommitted in-progress work that prod already includes. A worktree sits at a clean commit and would ship a bundle missing that work, wiping it from production. The prebuild guard catches this — do not bypass it. (We've lost work twice this way already.)

Gotchas — do NOT

pitfalls

A checklist of "things that look fine but blow up later". Most of these are also in CLAUDE.md; treat that file as the living version.

  • Don't use the any type. Define a proper interface in src/shared/types/.
  • Don't add npm dependencies without discussing first. Bundle size matters; React 19 + Tailwind 4 + Recharts is already chunky.
  • Don't modify Base44 entity schemas casually — unknown fields are silently dropped.
  • Don't put React hooks after conditional return statements (React error #300/#310). Especially in GameDetail.tsx and ChessDataContext.tsx.
  • Don't filter entities server-side by created_by_id — legacy records don't have that field.
  • Don't call runBatchAnalysis() directly — always go through queueForAnalysis() in ChessDataContext.
  • Don't use CSS variables (var(--chess-*)) in share-card components — html2canvas can't resolve them. Use SHARE_COLORS hex constants.
  • Don't assume entity.list() returns everything — Base44 caps at 5,000 records per call.
  • Don't trust tsc -b's exit code for "build succeeded" — check the dist/ output hash instead.
  • Don't deploy from a worktree under any circumstance.
  • Don't write to localStorage without considering the iOS 5 MB quota. That's the most common cause of mobile-Safari white-screen crashes.
  • Don't create extra README / planning files. The codebase has CLAUDE.md + this handoff — that is the documentation.
